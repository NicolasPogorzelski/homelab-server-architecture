# LXC260 – PostgreSQL Platform

This document describes the centralized PostgreSQL platform container (LXC260).

## Purpose

- Central database service for multiple application consumers
- Consistent hardening across all tenants
- Centralized backup to dedicated storage
- Strict tenant separation (one database + one user per service)

## Runtime Characteristics

- CTID: 260
- Hostname: postgres
- OS: Debian 12 (bookworm)
- Cores: 2
- RAM: 1024 MB
- Swap: 512 MB
- Root disk: 8 GB (local-lvm)
- Unprivileged: yes
- Nesting: enabled
- Onboot: yes
- Startup order: 2 (after storage, 20s delay)

## Mount Points

- `mp0`: `/mnt/aux1TB/postgres` → `/var/lib/postgresql` (PostgreSQL runtime data on auxiliary disk)
- `mp1`: `/mnt/smb/postgres-backups` → `/mnt/backups` (database backups on MergerFS — separate failure domain from runtime data)

Design rationale: Runtime data uses local block storage for ACID guarantees (fsync, WAL).
Backups use SMB/MergerFS because durability requirements differ — backup files are write-once
and tolerate higher latency. This separates the failure domains: a disk failure affecting
runtime data does not destroy backups, and vice versa.

## Installed Services

- PostgreSQL 15 (`127.0.0.1:5432` + Tailscale IP `:5432`)
- `postgres_exporter` v0.19.1 (systemd service, `<tailscale-ip-lxc260>:9187`) — exposes `pg_stat_*` metrics for Prometheus on CT200

PostgreSQL listens on loopback (local admin) and its Tailscale IP (remote consumer access).
It does NOT listen on `0.0.0.0` or any LAN interface.

### PostgreSQL Configuration Highlights

- Encoding: UTF-8
- Data checksums: enabled
- `listen_addresses`: `127.0.0.1, <tailscale-ip-lxc260>`
- `fsync`: on
- `synchronous_commit`: on

### pg_hba.conf (Hardened)

```
# Local admin (Unix socket, peer auth)
local   all             postgres                                peer

# Loopback (SCRAM auth for local tooling)
host    all             postgres        127.0.0.1/32            scram-sha-256

# --- Service allowlist (add per-service entries here) ---

# Default deny (everything else rejected)
host    all             all             0.0.0.0/0               reject
host    all             all             ::/0                    reject
```

New consumers are onboarded by inserting `hostssl` entries above the default deny block.
See: [PostgreSQL Platform Service](../services/postgresql-platform.md)

## Network / Tailscale

- Tailscale hostname: postgres
- Tag: `tag:database`
- Accessible by: `tag:admin` (all ports), consumers via explicit ACL rules (port 5432 only)

### Tailscale Serve

Not applicable. PostgreSQL uses a binary protocol (not HTTP), so Tailscale Serve
cannot proxy it. Consumers connect directly to the Tailscale IP on port 5432.

## Security / Exposure

- PostgreSQL binds to loopback + Tailscale IP only
- No LAN exposure
- No public exposure / no router port forwarding
- Remote access provided exclusively via Tailscale (identity-based overlay)
- Network policy enforced via Tailscale ACL (node tags + ACL JSON)
- Additional authentication layer via pg_hba.conf (independent from ACL)
- See: [docs/platform/tailscale-acl.md](../platform/tailscale-acl.md)

Access requires passing ALL layers:

    Tailscale ACL (network identity)
          ↓
    Tailscale-only bind (no LAN path)
          ↓
    pg_hba.conf (database-level allowlist)
          ↓
    PostgreSQL role permissions (per-tenant authorization)

See: [PostgreSQL Platform Service](../services/postgresql-platform.md) for the full 4-layer model.

## Failure Impact

If LXC260 becomes unavailable:

- All dependent services lose database connectivity
- Application startup may fail for DB-dependent services
- Existing connections are terminated
- No data loss if WAL and fsync guarantees were intact before failure
- Recovery priority: restore database service before restarting application containers
