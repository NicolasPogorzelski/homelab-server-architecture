# VM102 – Storage

This document describes the dedicated storage VM (VM102).

VM102 is the single source of truth for all persistent data in the infrastructure.

## Purpose

- SnapRAID parity-based data protection
- MergerFS unified namespace across data disks
- Samba exports with per-service identity segmentation
- Centralized storage for all service containers and VMs

## Runtime Characteristics

- VMID: 102
- Hostname: Storage
- OS: Debian 12 (bookworm)
- Cores: 2 (1 socket)
- RAM: 4096 MB
- System disk: 16 GB (local-lvm, virtio-scsi-single)
- SCSI controller: virtio-scsi-single
- Guest agent: enabled
- Onboot: yes
- Startup order: 1 (first after hypervisor, 30s delay)

## Disk Topology

### System Disk

- `scsi0`: local-lvm – OS + swap

### Data Disks (SnapRAID)

- `scsi1`–`scsi5`: 5 data disks (disk01–disk05)

### Parity Disk

- `scsi6`: 1 parity disk (must be >= largest data disk)

### Auxiliary Disks (non-parity)

- `scsi7`: aux-disk
- `scsi8`: aux01tb

All data and parity disks are passed through via `/dev/disk/by-id/` on the hypervisor for deterministic mapping.
Exact models and capacities are documented offline.

## Monitoring

- Node Exporter v1.11.1 (systemd binary, `/etc/systemd/system/node_exporter.service`)
- Binds to Tailscale IP only (`--web.listen-address=<tailscale-ip-vm102>:9100`)
- Textfile collector enabled (`--collector.textfile.directory=/var/lib/node_exporter/textfile_collector`)
- Textfile metrics written by `/usr/local/sbin/snapraid-maintenance.sh`:
  - `snapraid_sync.prom` — timestamp of last successful sync
  - `snapraid_scrub.prom` — timestamp of last successful scrub
- Cron jobs (`/etc/cron.d/snapraid`):
  - `snapraid sync` — daily at 23:00
  - `snapraid scrub` — monthly, 1st of each month at 20:00
- Prometheus alert rules: `SnapRAIDSyncStale` (>26h), `SnapRAIDScrubStale` (>32d)

## Time Synchronization

- `chrony` keeps the system clock synchronized (verify: `chronyc tracking`, `chronyc sources`, `timedatectl`).
- Why it matters here: a storage node's clock underpins SnapRAID change detection (timestamp + size based), backup correctness, the `SnapRAIDSyncStale` alert math, and cross-node log correlation during incidents.
- Before 2026-06-07 no time daemon was installed (`timedatectl`: `NTP service: n/a`, `synchronized: no`); the clock relied on RTC/hypervisor only and drifted. `chrony` installed + enabled; `timedatectl` now reports `synchronized: yes`.
- Originally installed ad-hoc; codified 2026-06-08 into the fleet-wide `chrony` role (`ansible/roles/chrony/`, applied to `vms`).
- LXCs do not run their own time daemon — they inherit the Proxmox host clock; this concern applies to VMs (vm100 was already synced, vm102 was the gap).

## Storage Stack

- MergerFS pool at `/mnt/mergerfs` (unified view of disk01–disk05 + aux-disk temporary capacity bridge)
- SnapRAID with single parity (manual sync + scrub)
- Samba exports with per-service users and least-privilege enforcement
- See: [Storage Design](../platform/storage-design.md)
- See: [Samba Configuration](../platform/samba.md)

## Network / Tailscale

- Tailscale hostname: storage
- Tag: `tag:storage`
- Accessible by: `tag:admin` (all ports), `tag:tier0` (all ports), `tag:tier1` (port 445), `tag:tier2` (port 445), `tag:gaming` (port 445)

### Tailscale Serve

Not applicable. This node exposes only SMB (port 445) for storage access. No web services.

## Security / Exposure

- Samba (port 445) is the only service exposed to other nodes
- SMB3 only, mandatory signing
- No public exposure / no router port forwarding
- Network policy enforced via Tailscale ACL (node tags + ACL JSON)
- See: [docs/platform/tailscale-acl.md](../platform/tailscale-acl.md)

## Failure Impact

VM102 represents a single storage failure domain.

If VM102 becomes unavailable:

- All dependent services lose access to their storage paths
- Containers may start but fail due to missing mounts
- Monitoring should detect mount and service degradation
- No data loss if disks are intact (recovery is procedure-based)

High availability is not implemented; recovery is procedure-based.

## Configuration Management

Managed via Ansible from LXC250.

- **Playbook:** `ansible/playbooks/apt-upgrade.yml` — rolling apt upgrade (`serial: 1`)
- **Role:** `ansible/roles/node_exporter/` — deploys node_exporter binary + systemd unit
- **Role:** `ansible/roles/ssh-hardening/` — `PasswordAuthentication no`, `PermitRootLogin no`
- **Role:** `ansible/roles/chrony/` — installs `chrony`, ensures started + enabled (time sync)
- **Note:** `PermitRootLogin` was explicitly set to `yes` before 2026-05-28 hardening — remediated by `ssh-hardening` role
- **SSH access model:** primary admin path is the `ansible` user (key-based, NOPASSWD sudo). Break-glass fallback: the `storage` user's `authorized_keys` also holds the admin workstation key (`desktop-cachyos`), added ad-hoc 2026-06-07, codified 2026-06-08 via the `breakglass` role (`ansible/roles/breakglass/`, `breakglass_user: storage`). `PasswordAuthentication no`, so key presence is the only access lever. Legacy keys present on `storage` (`root@server`, `fedora-notebook`) are flagged for review/cleanup.
- **Time sync:** `chrony` installed (see [Time Synchronization](#time-synchronization)) via the `chrony` role.
