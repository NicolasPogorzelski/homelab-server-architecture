# VM102 – Storage

This document describes the dedicated storage VM (VM102).

VM102 is the single source of truth for all persistent data in the infrastructure.

## Purpose

- SnapRAID parity-based data protection
- MergerFS unified namespace across data disks
- Samba exports with per-service identity segmentation
- Centralized storage for all service containers and VMs

## Runtime Characteristics

- VMID: 102
- Hostname: Storage
- OS: Debian 12 (bookworm)
- Cores: 2 (1 socket)
- RAM: 4096 MB
- System disk: 16 GB (local-lvm, virtio-scsi-single)
- SCSI controller: virtio-scsi-single
- Guest agent: enabled
- Onboot: yes
- Startup order: 1 (first after hypervisor, 30s delay)

## Disk Topology

### System Disk

- `scsi0`: local-lvm – OS + swap

### Data Disks (SnapRAID)

- `scsi1`–`scsi5`: 5 data disks (disk01–disk05)

### Parity Disk

- `scsi6`: 1 parity disk (must be >= largest data disk)

### Auxiliary Disks (non-parity)

- `scsi7`: aux3TB
- `scsi8`: aux1TB

All data and parity disks are passed through via `/dev/disk/by-id/` on the hypervisor for deterministic mapping.
Exact models and capacities are documented offline.

## Storage Stack

- MergerFS pool at `/mnt/mergerfs` (unified view of disk01–disk05)
- SnapRAID with single parity (manual sync + scrub)
- Samba exports with per-service users and least-privilege enforcement
- See: [Storage Design](../platform/storage-design.md)
- See: [Samba Configuration](../platform/samba.md)

## Network / Tailscale

- Tailscale hostname: storage
- Tag: `tag:storage`
- Accessible by: `tag:admin` (all ports), `tag:tier0` (all ports), `tag:tier1` (port 445), `tag:tier2` (port 445)

## Security / Exposure

- Samba (port 445) is the only service exposed to other nodes
- SMB3 only, mandatory signing
- No public exposure / no router port forwarding
- Network policy enforced via Tailscale ACL (node tags + ACL JSON)
- See: [docs/platform/tailscale-acl.md](../platform/tailscale-acl.md)

## Failure Impact

VM102 represents a single storage failure domain.

If VM102 becomes unavailable:

- All dependent services lose access to their storage paths
- Containers may start but fail due to missing mounts
- Monitoring should detect mount and service degradation
- No data loss if disks are intact (recovery is procedure-based)

High availability is not implemented; recovery is procedure-based.
