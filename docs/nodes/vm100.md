# VM100 Services (GPU / Media)

## Purpose

- GPU-accelerated media streaming (Jellyfin with NVIDIA hardware transcoding)
- Audiobook hosting (Audiobookshelf)
- Local LLM inference backend (Ollama, backup node for AI stack)
- Primary media consumption endpoint for LAN and Tailnet clients

This document describes the current containerized media services running on VM100 (Compute/GPU VM).

## Jellyfin

- Deployment: Docker Compose (`/opt/docker/jellyfin/docker-compose.yml` on VM100)
- GPU acceleration: enabled (`gpus: all` + NVIDIA environment variables)
- Runs as non-root user (`user: 1000:1000`)
- Persistent application data:
  - `${JF_CONFIG}` → `/config`
  - `${JF_CACHE}` → `/cache`
  - `${JF_METADATA}` → `/metadata`
- Media mounts (read-only, least privilege):
  - `${JF_MEDIA_FILME}` → `/media/Filme:ro`
  - `${JF_MEDIA_SERIEN}` → `/media/Serien:ro`
- Port exposure: `8096/TCP` (LAN)
- See: [Jellyfin Service](../services/jellyfin.md)

## Audiobookshelf

- Deployment: Docker Compose (`/opt/docker/audiobookshelf/docker-compose.yml` on VM100)
- Media mounts (read-only, least privilege):
  - `${ABS_AUDIOBOOKS_ROOT}` → `/audiobooks:ro`
  - `${ABS_PODCASTS_ROOT}` → `/podcasts:ro`
- Port exposure: `13378/TCP` (LAN)
- See: [Audiobookshelf Service](../services/audiobookshelf.md)

## Ollama

- Installation: native systemd service
- Override: `/etc/systemd/system/ollama.service.d/override.conf`
- Bind address: Tailscale IP only (`<tailscale-ip-vm100>:11434`)
- Model: `qwen3:8b-q4_K_M`
- Role: backup inference backend for OpenWebUI (CT230)
- See: [Ollama Service](../services/ollama.md)

## Storage Mount Strategy (Reboot-Safe)

Media paths on VM100 are provided via **systemd automount** units (fstype `autofs`) to ensure reboot-safe startup behavior:
- Jellyfin media path (referenced by `${JF_MEDIA_*}` variables)
- Audiobookshelf media path (referenced by `${ABS_*_ROOT}` variables)

Note: A legacy `media-mergerfs.mount` unit exists on VM100 but is not used for service mounts in the current state.

## Aux1TB Storage Status (2026-06-25)

VM100's second disk — the `jellyfin-data` image (`scsi1`, an ext4 `.raw` stored on
the Proxmox host's Aux1TB) — was hosted on the failing Aux1TB disk. Aux1TB suffered
unrecoverable medium errors and is being decommissioned (see
[KE-13](../platform/known-errors.md#ke-13-aux1tb-physical-disk-failure-medium-errors)).

- **Jellyfin** config/cache/metadata (`${JF_CONFIG}`/`${JF_CACHE}`/`${JF_METADATA}`
  → `/mnt/vm-data/jellyfin`) lived on this image and was **rescued read-only**
  (`jf-jellyfin.tar`, including `jellyfin.db` + artwork). No re-scrape needed.
- **Audiobookshelf** data (`${ABS_DATA_ROOT}` = `/opt/docker/audiobookshelf`) lives
  on the VM100 **root** disk (local-lvm), **not** on Aux1TB — unaffected; backed up
  as a precaution.
- VM100 and the Jellyfin stack are **down** until the jellyfin-data image is
  relocated onto healthy storage. Full context:
  [incident write-up](../platform/incidents/2026-06-25-aux1tb-failure-and-recovery.md).

## Network / Tailscale
- Tailscale hostname: gpu-vm
- Tag: `tag:tier2`
- Accessible by: `tag:admin` (all ports), `tag:tier0` (all ports), `tag:tier2` (all ports), `tag:ai-stack` (port 11434), `tag:client` (ports 443, 8096, 13378), `tag:untrusted` (ports 443, 8096, 13378)

### Tailscale Serve

Not in use. Jellyfin and Audiobookshelf bind to `0.0.0.0` for LAN streaming performance.
Remote access uses the Tailscale IP directly (WireGuard-encrypted, no TLS hostname).

See: [Loopback + Tailscale Serve ADR](../decisions/loopback-tailscale-serve.md) — section "Documented Exceptions"

## Security / Exposure Notes (Trade-off)

- No public reverse proxy and no router port-forwarding.
- Default remote access path is Tailscale (identity-based overlay).
- Network policy enforcement is defined via Tailscale ACL (tags + JSON policy).
  See: [docs/platform/tailscale-acl.md](../platform/tailscale-acl.md)

### Intentional LAN Exposure

- Jellyfin (`8096/TCP`) and Audiobookshelf (`13378/TCP`) are intentionally reachable on the LAN.
- This is a documented performance trade-off (bandwidth/latency optimization and reduced transcoding load).
- LAN exposure must remain limited to these specific ports only.

## Failure Impact

If VM100 becomes unavailable:

- No media streaming (Jellyfin, Audiobookshelf)
- No GPU-accelerated workloads
- Storage layer (VM102) is NOT affected
- No data loss (media is read-only from SMB storage)
- Recovery: restart VM, verify mounts, confirm Docker containers are running

## Time Synchronization

- `chrony` keeps the system clock synchronized (verify: `chronyc tracking`, `chronyc sources`, `timedatectl`).
- Why it matters here: cross-node log correlation during incidents (see KE-8), and a consistent time source across the VM fleet.
- Before 2026-06-08 vm100 used `systemd-timesyncd` (the Debian default) and was already synced — it was never the gap. The `chrony` role replaced timesyncd fleet-wide for a single, consistent time daemon across both VMs; the `chrony` package `Conflicts:` with `systemd-timesyncd`, so apt removed timesyncd on install.
- LXCs do not run their own time daemon — they inherit the Proxmox host clock; this concern applies to VMs only.

## Configuration Management

Managed via Ansible from LXC250.

- **Playbook:** `ansible/playbooks/apt-upgrade.yml` — rolling apt upgrade (`serial: 1`)
- **Role:** `ansible/roles/node_exporter/` — deploys node_exporter binary + systemd unit
- **Role:** `ansible/roles/ssh-hardening/` — `PasswordAuthentication no`, `PermitRootLogin no`
- **Role:** `ansible/roles/chrony/` — installs `chrony`, ensures started + enabled (replaced `systemd-timesyncd`)
- **Role:** `ansible/roles/breakglass/` — deploys the admin break-glass pubkey (`desktop-cachyos`) to the native `gpu` user (`breakglass_user: gpu`)
- **SSH access model:** primary admin path is the `ansible` user (key-based, NOPASSWD sudo). Break-glass fallback: the `gpu` user's `authorized_keys` holds the admin workstation key (`desktop-cachyos`), codified 2026-06-08 via the `breakglass` role. `PasswordAuthentication no`, so key presence is the only access lever. VMs need an SSH break-glass because out-of-band `qm guest exec` depends on the qemu-guest-agent; LXCs are reachable via `pct exec` and need none.
- **Role:** `ansible/roles/docker-compose-update/` (playbook `ansible/playbooks/docker-compose-update.yml`, group `docker`) — updates this node's two Compose stacks (`/opt/docker/audiobookshelf`, `/opt/docker/jellyfin`) via `docker_compose_v2`: `pull: always` + `recreate: auto` (idempotent — recreates only on image/config change). The list var `compose_projects` in `host_vars/vm100.yml` iterates both, one stack at a time.

See: [Ansible Platform](../platform/ansible.md)
