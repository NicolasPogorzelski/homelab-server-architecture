# LXC240 – Vaultwarden

This document describes the Vaultwarden container (LXC240).

## Purpose

- Self-hosted password manager (Bitwarden-compatible)
- Security-critical service (secrets tier)

## Runtime Characteristics

- CTID: 240
- Hostname: vaultwarden
- OS: Debian 12 (bookworm)
- Cores: 2
- RAM: 1024 MB
- Swap: 512 MB
- Root disk: 8 GB (local-lvm)
- Unprivileged: yes
- Nesting: enabled
- Onboot: yes
- Startup order: 4 (after storage + monitoring + nextcloud, 20s delay)

## Mount Points

- `mp0`: `/mnt/smb/vaultwarden` → `/opt/vaultwarden` (persistent data on SMB storage)

## Network / Tailscale

- Tailscale hostname: vaultwarden
- Tag: `tag:tier1`
- Accessible by: `tag:admin` (all ports), `tag:tier0` (all ports), `tag:tier1` (all ports), `tag:client` (port 443)

## Security / Exposure

- Classified as tier 1 (security-critical)
- Loopback-only binding (`127.0.0.1:8080`)
- No LAN exposure
- No public exposure / no router port forwarding
- Remote access provided exclusively via Tailscale (identity-based overlay)
- Network policy enforced via Tailscale ACL (node tags + ACL JSON)
- See: [docs/platform/tailscale-acl.md](../platform/tailscale-acl.md)

## Failure Impact

If LXC240 becomes unavailable:

- No access to password vault
- Clients with cached credentials continue to work temporarily
- SQLite database and encryption keys become inaccessible
- Recovery: recreate LXC, remount storage, redeploy Docker container
- Backups of database and key material are critical
